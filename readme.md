# AI Proto
claude codeにAIを作らせるセットです。
詳細は contracts_clm_code_prototyping_context.md に記載していますが、以下に利用方法を抜粋します。

## そのまま使えるプロンプト雛形

### A. 新規画面を作るとき

```text
ContractS CLM の既存 UI ルールに沿って、PCブラウザ向けの業務 SaaS 管理画面をコードベースで 1 画面実装してください。

あなたは B2B SaaS のシニア UI デザイナー兼フロントエンド実装者として振る舞ってください。
見た目だけでなく、ユーザーが何を確認し、何を判断し、何を実行する画面かが明確に伝わる構造を優先してください。
必要であれば、画面に不足している機能や情報ブロックも提案し、自然な形で構成に反映してください。

前提:
- HTML / CSS / JS ベースで実装可能なプロトタイプにする
- 日本語UIにする
- B2B SaaS の管理画面として自然な密度と情報設計にする
- primary action は原則 1 つにする
- empty / error / disabled を確認できるようにする
- ラベル、値、状態、補助情報を視覚的に分離する
- コード上の意味を持たない装飾専用要素は増やさない

画面タイプ:
{screen_type}

ユーザーの目的:
{user_goal}

主要タスク:
{primary_tasks}

表示する情報:
{display_fields}

ユーザー操作:
{user_actions}

必要な状態:
{required_states}

制約:
{constraints}

議論したい論点:
{design_questions}

出力してほしいもの:
- 画面コード
- 主要な状態差分
- 必要に応じた追加機能の提案
```

### B. 既存画面を改善するとき

```text
既存の ContractS CLM 画面を、コードベースのプロトタイプとして改善してください。
既存画面のキャプチャを添付するので、その画面構造と現状UIを出発点にしてください。

目的は見た目の調整だけではなく、情報の主従、主要操作、状態の分かりやすさを改善することです。
あなたは B2B SaaS のシニア UI デザイナー兼フロントエンド実装者として、一般的な情報設計と状態設計のセオリーに沿って判断してください。

改善方針:
- 添付した既存画面キャプチャの情報構造を読み取り、維持すべき点と改善すべき点を整理する
- first glance で主目的が分かるようにする
- primary action を迷わせない
- ラベル、値、状態、補助情報の競合をなくす
- 必要に応じて不足している empty / error / disabled を補う
- 過度な装飾ではなく、構造改善で解決する

入力:
- 既存画面キャプチャ
- 必要に応じて現行コードまたは補足説明

改善対象:
{target_screen}

現状の課題:
{current_problems}

維持したいもの:
{must_keep}

追加したい要素:
{requested_additions}

必要な状態:
{required_states}

出力してほしいもの:
- 改善後のコード
- 何をどう改善したか
- 追加した状態や機能の理由
```

### C. 機能議論をしながら試作するとき

```text
ContractS CLM の新しい画面案を、コードベースのプロトタイプとして提案してください。

この依頼では、完成した見た目を出すだけでなく、「B2B SaaS の画面としてこの機能や情報整理が必要では」という観点も含めて提案してください。
あなたは B2B SaaS のシニア UI デザイナー兼フロントエンド実装者として、業務フロー、判断、例外処理、状態管理の観点から不足を補ってください。

前提:
- HTML / CSS / JS ベースで確認できるプロトタイプにする
- 主要フローと主要状態が分かることを優先する
- 提案は業務上の意味で説明できるものだけにする
- 余計な装飾ではなく、情報設計と操作設計で改善する

テーマ:
{theme}

解決したい課題:
{problem_to_solve}

現時点で想定している機能:
{assumed_features}

まだ議論したい点:
{open_questions}

必要な状態:
{required_states}

出力してほしいもの:
- コードベースの画面案
- 追加した方が良い機能や情報ブロック
- その提案理由
- 優先度が高い論点
```

### D. コンポーネント仕様を固めたいとき

```text
以下の UI 要素を、ContractS CLM のコードベース・プロトタイプで再利用できるコンポーネントとして定義してください。

あなたは B2B SaaS のシニア UI デザイナー兼フロントエンド実装者として、見た目だけでなく責務、状態、使い分けが明確になるように整理してください。

対象コンポーネント:
{component_name}

使いたい場面:
{use_cases}

必要な状態:
{required_states}

制約:
{constraints}

出力してほしいもの:
- コンポーネントの責務
- 必要な props / バリエーション相当の整理
- 状態差分
- コードベースの試作実装方針
```

## プロンプト作成時の補足ルール

- 画面タイプだけでなく、ユーザーの目的と主要タスクを必ず含める
- 見た目要件だけでなく、必要な状態を明示する
- 「何を議論したいか」を入れると、機能提案まで返りやすくなる
- 「維持したいもの」を書くと、不要な破壊的変更を防ぎやすい
- 「出力してほしいもの」を明示すると、コードだけで終わらず意図した粒度の回答になりやすい
- 既存画面の改善では、対象画面のキャプチャを添えると、維持すべき構造と改善点を切り分けやすい
