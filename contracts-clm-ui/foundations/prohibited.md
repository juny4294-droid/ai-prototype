# Prohibited

## Purpose

ContractS CLM の UI において、
業務用途としての可読性・信頼性・一貫性を損なう設計を防ぐ。

---

## 1. Visual Style Constraints（スタイル禁止）

本プロダクトでは以下のビジュアル表現は採用しない。

- グラデーション
  - 理由: 情報階層が曖昧になる
- ネオンカラー
  - 理由: 業務UIとしての信頼性を損なう
- ガラスモーフィズム（背景透過・ぼかし）
  - 理由: 可読性が低下する
- 過剰なシャドウ（立体強調）
  - 理由: UIの意味構造ではなく装飾が強くなる
- 3D表現・イラスト的装飾UI
  - 理由: 実務画面としての一貫性が崩れる

---

## 2. Layout Anti-patterns（構造の失敗）

- 1画面に複数の primary action を配置する
- 情報の主従が視覚的に判別できない
- テーブルに過剰な列を詰め込む
- 詳細画面で本文と補助情報の境界が曖昧
- カード分割に意味がなく、単なる装飾になっている
- フィルタ → 一覧 → ページネーション の順序を崩す
- サイドバーや補助情報が主コンテンツを圧迫する

---

## 3. Interaction Anti-patterns（操作の失敗）

- 自動保存と手動保存が混在する
- 保存責務が不明確（いつ確定するか不明）
- 破壊操作に confirmation がない
- row click と button click が競合する
- disabled の理由が表示されない
- hover / focus / active の違いがない

---

## 4. Accessibility Violations（アクセシビリティ違反）

- 色だけで状態（必須 / エラー / 完了）を表現する
- 読めないサイズの日本語（10px未満）
- コントラスト不足（背景と文字が近い）
- エラーが短時間で消える（重要なもの）
- フォーカス状態が視認できない

---

## 5. Judgment Criteria（判定基準）

以下のいずれかに該当する場合、そのUIは不適切と判断する。

- 業務画面として説明できない
- 情報の主従が1秒で判断できない
- primary action が迷う
- 次に何をすべきか分からない
- 状態（エラー・成功・無効）が直感的に分からない

---

## 6. Enforcement Rules（LLM適用ルール）

LLMがUIを生成する際は以下を強制する。

- 上記の禁止項目を一切含めない
- primary action は1つのみ生成する
- 情報階層を typography / spacing / layout で表現する
- state（hover / focus / error / disabled）を必ず定義する
- 見た目よりも構造の明確さを優先する
