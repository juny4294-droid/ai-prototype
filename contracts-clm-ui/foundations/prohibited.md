# Prohibited

## Purpose

ContractS CLM の UI において、
業務用途としての可読性・信頼性・一貫性を損なう設計を防ぐ。

---

## 1. Visual Style Constraints（スタイル禁止）

本プロダクトでは以下のビジュアル表現は採用しない。

- グラデーション
  - 理由: 情報階層が曖昧になる
- ページ背景の角や帯に装飾目的のグラデーションを入れる
  - 理由: 一覧・設定画面のベース面として不要で、情報より装飾が先に立つ
- ネオンカラー
  - 理由: 業務UIとしての信頼性を損なう
- ガラスモーフィズム（背景透過・ぼかし）
  - 理由: 可読性が低下する
- 過剰なシャドウ（立体強調）
  - 理由: UIの意味構造ではなく装飾が強くなる
- 3D表現・イラスト的装飾UI
  - 理由: 実務画面としての一貫性が崩れる

---

## 2. Layout Anti-patterns（構造の失敗）

- 1画面に複数の primary action を配置する
- 情報の主従が視覚的に判別できない
- 重要度の異なる情報を同じ強さで並べる
- テーブルに過剰な列を詰め込む
- 詳細画面で本文と補助情報の境界が曖昧
- カード分割に意味がなく、単なる装飾になっている
- 一覧画面で要件にないダッシュボード要約カードを先頭に挿入する
- 詳細検索をページ構造ではなく装飾カードとして浮かせる
- フィルタ → 一覧 → ページネーション の順序を崩す
- サイドバーや補助情報が主コンテンツを圧迫する
- 見出し直上に意味の薄い英字ラベルを置いて情報階層を増やす
- 同じ意味の情報を要約と詳細で重複表示する
- 指定のない左右並びカードで情報を分割する
- 段組の左端・上端が揃わず、title / meta / action / divider の基準線が揺れる
- card、panel、table、viewer が理由なくぴったり接して境界が消える
- section note、legend、badge group と、その直下の card / tree / list を密着させる
- 既存 shell の gutter、rail、panel 幅を無視して、画面ごとに別寸法を発明する
- compact に見せるために padding を削り、可読性より面の圧縮を優先する
- 「v0っぽく見せる」ためだけの summary card、KPI card、hero block を自動追加する
- form 内で根拠なく 2 カラム配置を行い、読み順を崩す
- panel 内で link、button、icon button のサイズや基準線が揃わない

---

## 3. Interaction Anti-patterns（操作の失敗）

- 自動保存と手動保存が混在する
- 保存責務が不明確（いつ確定するか不明）
- 破壊操作に confirmation がない
- row click と button click が競合する
- disabled の理由が表示されない
- hover / focus / active の違いがない

---

## 4. Accessibility Violations（アクセシビリティ違反）

- 色だけで状態（必須 / エラー / 完了）を表現する
- 読めないサイズの日本語（10px未満）
- コントラスト不足（背景と文字が近い）
- エラーが短時間で消える（重要なもの）
- フォーカス状態が視認できない

---

## 5. Judgment Criteria（判定基準）

以下のいずれかに該当する場合、そのUIは不適切と判断する。

- 業務画面として説明できない
- 情報の主従が1秒で判断できない
- primary action が迷う
- 次に何をすべきか分からない
- 状態（エラー・成功・無効）が直感的に分からない
- コンテンツ内の重要度がぱっと見で分からない

---

## 6. Enforcement Rules（LLM適用ルール）

LLMがUIを生成する際は以下を強制する。

- 上記の禁止項目を一切含めない
- primary action は1つのみ生成する
- 情報階層を typography / spacing / layout で表現する
- state（hover / focus / error / disabled）を必ず定義する
- 見た目よりも構造の明確さを優先する
- タイトルやセクション見出しは日本語を正式表記とし、英字の補助ラベルを自動追加しない
- ラベル、値、状態、補助情報の役割が混ざらないように分離する
- まず読む情報、次に判断する情報、最後に補助情報の順で並べる
- 段組、余白、divider、padding は既存 foundations の固定値を優先し、任意の見た目調整を後から足さない
- card や panel を追加する前に、その block が `状態把握 / 判断 / 実行` のいずれに必要かを説明できること
