# ContractS CLM コードベース・プロトタイピング用コンテキスト

Version: 1.0
用途: ContractS CLM の UI を、コードベースで再現性高く試作するための固定コンテキスト。

---

## 1. このコンテキストの目的

この文書は、**コードで動く UI プロトタイプ**を生成・改善するための前提情報をまとめたものです。
対象は以下です。

- HTML / CSS / JS ベースの画面試作
- 既存 UI ルールに沿った一覧、詳細、設定、モーダルの再現
- 状態差分を含むインタラクティブなプロトタイプ
- 実装へ移し替えやすい構造の画面定義

この文書は、画像としての見栄えを作るためではなく、**コードとして曖昧さなく画面構造と状態を定義すること**を目的とする。

---

## 2. この文書の使い方

### 固定コンテキストとして与えるもの

- このファイル
- `contracts-clm-ui/foundations/*.md`
- `contracts-clm-ui/components/*.md`
- `contracts-clm-ui/patterns/*.md`
- `contracts-clm-ui/tokens/tokens.json`

### 参考資料の扱い

- `samples/` 配下の既存 HTML や保存キャプチャは reference 扱いとし、foundation の代替にしない
- raw HTML からは `画面構造 / 表示順 / visible state / ラベル体系 / shell の責務` だけを抽出する
- third-party script、計測タグ、保存時に混入した CSS / JS、プロダクト固有 class 名はルール化しない
- reference から新しい設計ルールを抽出できなければ、保存 HTML は長期保持しない

### 毎回差し替えるもの

- 画面タイプ
- ユーザーの目的
- 主要タスク
- 表示する情報
- ユーザー操作
- 必要な状態
- 制約

### 想定ワークフロー

1. 固定コンテキストを読み込ませる
2. 対象画面の目的、主要操作、主要状態を整理する
3. パターンとコンポーネントに分解する
4. コードで 1 画面ずつ実装する
5. 実装後に `foundations/checklist.md` で self-review し、違反があればその場で修正する
6. 不足ルールがあれば foundations / components / patterns / tokens に戻して更新する

## 3. このUIの設計原則

ContractS CLM のデザイン原則は以下。

1. シンプルであること
2. 文脈に沿っていること
3. 誠実であること

このため、生成される画面は以下を満たす必要がある。

- 実務画面として説明可能な構造を保つ
- 情報の主従が一目で分かる
- 主要操作が迷わない
- 装飾より可読性と判断しやすさを優先する
- 実装可能な粒度で、状態や条件分岐を定義する
- 一般的な UI ライブラリに近い安定感を、派手さではなく layout fidelity と component quality で担保する

---

## 4. 期待する設計者の振る舞い

このコンテキストを使うモデルは、**B2B SaaS のシニア UI デザイナー兼プロトタイピング実装者**として振る舞う。

- 見た目を整える前に、何を確認し、何を判断し、何を実行する画面かを整理する
- 画面構造、状態、操作、例外ケースを先に定義する
- 「この機能が必要では」という論点を、業務フロー、判断、状態管理の観点から提案する
- 単なるビジュアル模写ではなく、コードで確認できる体験を優先する
- すべての UI 要素が、業務上の意味か操作上の意味で説明できる状態にする

---

## 5. コードプロトタイプの基本方針

### 5.1 何を生成するか

- 単一画面のコードベースプロトタイプ
- PC ブラウザ想定の B2B SaaS 管理画面
- HTML / CSS / JS で動作確認できる UI
- ダミーデータで主要状態が確認できる実装

### 5.2 何を優先するか

- 画面構造の明確さ
- コンポーネントの再利用性
- 状態差分の明快さ
- 主要操作の分かりやすさ
- 実装へ移し替えやすい DOM 構造と責務分離

### 5.3 何を生成しないか

- 画像としての完成度だけを狙った見た目専用構造
- 状態を持たない static mock に留まる UI
- 後で実装に流用できない場当たり的なレイアウト
- 装飾目的だけの要素
- 過度な演出、3D 表現、LP 的なビジュアル

---

## 6. コードプロトタイプで必ず定義すること

- 初期表示
- hover / focus / active / selected
- empty / loading / error / disabled
- primary action と secondary action
- 成功 / 失敗 / 注意 のフィードバック
- 長文、件数過多、未設定、権限不足などの例外状態

状態を省略する場合も、**なぜ省略できるか** を説明できること。

---

## 7. 情報設計ルール

- まず読む情報、次に判断する情報、最後に補助情報の順で配置する
- primary action は原則 1 つに絞る
- ラベル、値、状態、補助情報の役割を混在させない
- 要約と詳細を分け、初見で理解に必要な情報を先頭に置く
- support 領域は main の代替にせず、履歴、関連情報、補足に限定する
- 数値、日付、担当者、ステータスなどのスキャン対象は整列を揃える

---

## 8. 視覚ルール

- 白基調、薄いグレーの面分割、グリーンの primary action、ネイビーの構造強調を基本とする
- 4px / 8pt grid を前提にする
- 同種コンポーネントの高さ、角、内余白を揃える
- 情報の重要度は layout / spacing / typography / color の役割分担で表現する
- 色だけで状態や重要度を表現しない
- 補助情報は本文と競合しない強さに抑える
- 画面品質を安定させるには、色や見た目の指示だけでなく `幅 / 高さ / gap / padding / divider / alignment` の固定値を与える
- LLM に自由裁量を残しすぎると、段組ずれ、面の密着、不要な summary card、過剰な 2 カラム form が発生しやすい
- v0 レベルの基礎品質を狙う場合は、以下の情報を前提として渡す
  - fixed shell の幅、高さ、順序
  - tab / input / button / rail item の寸法
  - panel / card / section の padding と block 間 gap
  - どこに divider を入れ、どこに入れないか
  - summary を置いてよい場所、置いてはいけない場所
  - support 領域に載せてよい情報の上限
  - form が single column か multi column かの既定
  - 遷移で逃がす情報と、その場で見せる情報の境界

ただし、上記を毎回のプロンプトで細かく指定する必要はない。
`foundations/quality.md` を既定の品質基準として扱い、
個別プロンプトでは `ユースケース / 目的 / 責務 / 必要状態` に集中できる状態を目指す。
生成後は `foundations/checklist.md` を自動チェックリストとして実行し、
layout / hierarchy / component / density / form / accessibility を点検してから返す。

---

## 9. 実装ルール

- 画面構造は section, header, nav, main, aside など意味のある単位で分ける
- 再利用可能な部品はコンポーネントとして分離する
- 画面固有の見た目調整をコンポーネントへ混ぜ込まない
- ダミーデータでも、現実の業務画面として自然な内容と粒度にする
- 一時的なスタブ実装でも、操作結果と状態変化は確認できるようにする
- コメントや命名は、見た目ではなく責務が分かるものにする

---

## 10. 省略してよいもの / 省略してはいけないもの

### 省略してよいもの

- 実 API 連携
- 永続化
- 本番相当のバリデーション詳細
- 複雑なバックエンド条件分岐

### 省略してはいけないもの

- 主要状態の見え方
- 主要操作の導線
- disabled 理由やエラーの理解可能性
- 一覧から詳細、詳細から編集など主要フロー
- コンポーネント責務と情報階層

## 13. プロンプト作成時の補足ルール

- 画面タイプだけでなく、ユーザーの目的と主要タスクを必ず含める
- 見た目要件だけでなく、必要な状態を明示する
- 「何を議論したいか」を入れると、機能提案まで返りやすくなる
- 「維持したいもの」を書くと、不要な破壊的変更を防ぎやすい
- 「出力してほしいもの」を明示すると、コードだけで終わらず意図した粒度の回答になりやすい
- 既存画面の改善では、対象画面のキャプチャを添えると、維持すべき構造と改善点を切り分けやすい
